JIT vs AOT — The Big Picture

Lesson 1 of ? — Mission: AOT Compilation

You know the compiler pipeline: source → parse → IR → optimize → codegen → machine code. What changes when you move that pipeline from runtime to build time? This lesson maps the two approaches side by side.

Two compilation strategies

Every compiler turns high-level code into machine code. The question is when.

StageJIT (Just-in-Time)AOT (Ahead-of-Time)
WhenDuring executionBefore execution (build time)
InputBytecode or IRSource or bytecode
OutputMachine code in memoryExecutable file on disk
DurationMust be fast (ms per method)Can be slow (minutes for whole program)
Info availableRuntime profile (which paths are hot)Static analysis only (unless PGO)
Key insight: In your JIT compiler, codegen happens while the user waits. In an AOT compiler, it happens while they build — so you can afford heavier optimization passes.

The two pipelines share the same middle (IR → optimize → codegen), but everything around it changes.

The closed-world assumption

This is the single most important idea in AOT compilation.

A JIT can assume anything might happen — new classes loaded, reflection calls, dynamic proxies. The JIT must guard every optimization with runtime checks (e.g., "is this virtual call still monomorphic?").

An AOT compiler makes a closed-world assumption: it statically analyzes every reachable class, method, and resource from the entry point. If something might be reachable, it gets compiled; if it's definitely not reachable, it's excluded. This lets the AOT compiler:

  • Inline virtual calls (it knows all possible implementation types)
  • Eliminate dead code entirely (smaller binary)
  • Statically allocate objects whose lifetime is known
Tradeoff: Reflection, dynamic class loading, and native interop break the closed world. AOT systems like GraalVM require explicit configuration (reflection config files) for anything dynamic.

The warmup gap

This is where the two strategies differ most sharply in practice:

MetricJITAOT
Startup timeSeconds (interpret + warmup)Milliseconds
Peak throughputExcellent (runtime PGO)~80-90% of JIT peak
Memory at peakHigher (JIT code cache + profiling data)Lower (precompiled, no profiling overhead)
Time to peakMinutes (warmup)Instant

A JIT's peak throughput is higher because it observes exactly which code paths are hot and can specialize without guesswork. But it pays for that with startup latency and memory overhead.

An AOT binary starts at full speed — there is no "warmup." It's immediately running compiled code. The code may be slightly less optimal, but it's optimal now.

Bridging the gap: Profile-Guided Optimization (PGO)

PGO is a two-phase AOT strategy:

  1. Instrumented build: Compile the program with profiling probes inserted
  2. Run on representative data: Execute the instrumented binary to collect profiles
  3. Optimized rebuild: Recompile using the collected profile data

This gives AOT some of the benefits of runtime profiling — but the profile is static. If the production workload differs from the training workload, the optimization can mispredict.

Your compiler in context

Your JIT compiler reads .class files, translates to LLVM IR, optimizes, and emits x86. Each stage maps to the AOT model like this:

Your JIT stageWhat AOT would change
Class loadingHappens at build time — not at runtime
Bytecode → IRSame translation, just earlier
LLVM optimization passesCan use -O3 or LTO (build time is cheap)
CodegenSame x86 emission, but to an object file
Method inliningAOT can inline more aggressively (closed-world knowledge)
DevirtualizationAOT can devirtualize all calls it can see

Check your understanding

Look at your compiler's pipeline (src/j2ll/). Ask yourself:

  • Which optimizations assume the program is still running? (profile data, hotness)
  • Which could run at build time with no change? (IR optimization passes)
  • What closed-world information isn't available to your JIT that an AOT compiler would have?
Tangible win: You can now look at any JIT compiler and identify the three things AOT changes — when compilation happens, how much optimization is affordable, and what assumptions the compiler can make.
