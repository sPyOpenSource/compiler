Numeric Types and the Stack

Lesson 2 of ? — Mission: AOT Compilation

In a high-level language, we think of a long as "just a bigger integer." But for a compiler writer, types are about memory layout and register constraints. In the JVM, this is most visible in how the operand stack is managed.

The Slot Model: 1 vs 2

The JVM operand stack is organized into "slots." A slot is typically 32 bits wide. Most types fit into a single slot, but the 64-bit types are special.

Java TypeJVM Slot SizeX86-64 Register MappingWidth
int, float1 SlotGPR (e.g., EAX) / XMM (e.g., XMM0)32-bit
long, double2 SlotsGPR (e.g., RAX) / XMM (e.g., XMM0)64-bit
reference1 SlotGPR (e.g., RAX)64-bit (native)
The Two-Slot Headache: When you push a long onto the stack, it occupies two consecutive slots. If you then push an int, it sits on top of those two. When popping a long, the compiler must ensure it pops both slots.

This is why your RuntimeStack implementation needs MODE_SLOT2. It tracks that a value is "wide," so that a pop() operation removes two 32-bit units of data from the stack.

Registers: GPRs vs. XMM

The CPU doesn't treat all numbers the same. They are handled by different "execution units" and stored in different register files:

  • GPRs (General Purpose Registers): Used for int and long. On x86-64, these are 64-bit registers (RAX, RBX, etc.). For 32-bit int, we use the lower half (EAX).
  • XMM Registers: Used for float and double. These are part of the SSE/AVX extensions. They are always at least 128 bits wide, but we use them to store single-precision (32-bit) or double-precision (64-bit) values.
Compiler Insight: When you see a bytecode like fadd (float add), your compiler must emit an x86 instruction like addss (Scalar Single-precision). For dadd (double add), it uses addsd (Scalar Double-precision).

Type Promotion (The Casts)

Because types are different sizes, the JVM uses explicit "cast" instructions to move data between them. These are not just "labels"—they are actual operations that change the bit representation.

  • i2l (int to long): Sign-extends a 32-bit value to 64-bit. (X86: movsxd)
  • l2i (long to int): Truncates the upper 32 bits. (X86: just use the lower register)
  • i2f (int to float): Converts an integer to IEEE 754 floating point. (X86: cvtsi2ss)
  • f2d (float to double): Expands the exponent and mantissa. (X86: cvtss2sd)

Tangible Win: Stack Tracing

Let's trace a small piece of bytecode. Try to determine the stack depth (in slots) after each instruction.

1. iconst_1       // Push int 1
2. lconst_0       // Push long 0
3. iadd           // ERROR!

Why is this an error? iadd expects two int values on top of the stack. But the top of the stack is a long (2 slots). The JVM specification requires that the types match the instruction. A long cannot be treated as two ints for an iadd operation.

Exercise: Imagine the stack is currently empty. 1. iconst_1 (1 slot) 2. lconst_0 (2 slots) 3. i2l (pops 1, pushes 2) 4. ladd (pops 2+2, pushes 2)
Final stack depth? 2 slots.
